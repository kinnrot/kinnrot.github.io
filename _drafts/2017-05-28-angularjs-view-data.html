--- layout: post title: AngularJS view data date: '2017-05-28T14:41:00.001-07:00' author: Chen Kinnrot tags: modified_time: '2017-05-28T14:41:17.092-07:00' blogger_id: tag:blogger.com,1999:blog-8908829452516573489.post-7433642921641996369 blogger_orig_url: http://kinnrot.blogspot.com/2017/05/angularjs-view-data.html --- Its a very common practice to load data from remote source and display it, there are so many ways of achieving this, The most common pattern I've ran into is:

1. ui router initialize a controller
2. controller calls some generic loadData method
3. loadData calls for remote data that hopefully is in a service
4. once data received view data initialized and user see stuff.

All of this is working and nice but this is not how the gods of angularjs wanted you to do it.

There are so many reasons why not to do it but lets try to keep it simple, so:

First thing,

The controller should not be responsible for loading the initial data
The controller should get the data as a dependency, why? all of the component approach really prefer the binding once/one way and it's a pain to bind once to something that does not exists yet (data did not load yet).

So who loads the data and when?
There is a very nice feature called resolve for ui-router. it allows you to define a set of promises that need to return before controller is loaded. when controller initialize it already has the data and can set it on the scope immediately, this way all bindings will work without any issues.
